# Sync — review rows made actionable

**Shipped:** 2026-06-02
**Commits:** PR #184 (`feat/sync-review-types-actionable`) — backend `f1be9b7` + review-fix `4e822d3`, extension `bf74505` + review-fix `84faeb4`, docs (this commit + `2e1b3f5`)

## What shipped

The extension's Sync drift report no longer surfaces dead-end "review" rows that offer no action *where an SBA-side action can be derived*. Sync is Kindoo-authoritative (it never writes SBA → Kindoo), so a `review`-severity row was a dead end: no Update Kindoo (authoritative) and no Update SBA either. This change derives an Update-SBA action wherever one is safe and meaningful, and keeps `review` (display-only) for the cases where one can't be.

Three buckets:

- **`extra-kindoo-calling`** (auto seat whose Kindoo Description lists calling(s) the seat lacks): reclassified `review → drift`. The action's user-facing label moved from **"Add to SBA seat"** to **"Update SBA"** to match the other Update-SBA buttons. The underlying behaviour — `syncApplyFix` with `code: 'extra-kindoo-calling'` de-dupes and appends the missing calling(s) to the seat's `callings[]` — is unchanged, and the dispatch `testId` stays `add-callings-sba`.
- **`kindoo-unparseable`** is split on the parser's blank-vs-present distinction (`parsed.segments.length === 0`):
  - **present-but-unparseable** (Description present but doesn't match `Scope (Calling)`): treated as a **church-wide (stake-scope) calling**, with the actionable "Update SBA" row **gated three ways in the detector** (refined by the code review, `84faeb4`): (A) **home site only** — on a foreign Kindoo site the row is suppressed entirely, since "apply to stake scope" is a home/stake concept (`isHomeSite`, off the same `activeSite` signal as the T-42 logic); (B) **Guest vs non-Guest** — a Guest (`UserRole === 2`) gets the actionable `drift` row, while a non-Guest (Manager / admin, `skipGrantReconciliation` true) gets a `review` row with **no** action, because an Update SBA would clobber a manager's seat; (C) **not already aligned** — for a home-site Guest the drift row fires only when the seat isn't already in the Update-SBA target state (`unparseableAligned`: `scope==='stake'` + calling-per-§6.1 matching the description, normalized). Once applied or already-matching, the row resolves and disappears on the next Sync run like every other drift code. On apply, the callable (`applyKindooUnparseable`, payload `{ memberEmail, calling }` = raw Description) sets `scope='stake'`, **clears `kindoo_site_id`** (stake-scope ⇒ home, spec §15), preserves `type`, and writes the calling per the §6.1 convention. For an **auto** seat it reaps the OLD scope's `importer_callings` and then, **iff** the calling matches a `give_app_access` **stake** template, writes a fresh `importer_callings['stake'] = [calling]` so the member keeps stake-scope app access — a bare template name like `Stake Clerk` reaches this path and *is* a real template; a non-template calling earns no new grant (old scope still reaped, access doc deleted when empty). The whole access-doc reshape is one coherent write (`writeStakeScopeAccessForUnparseable`, `tx.update` not `set merge`).
  - **blank Description** (empty / whitespace): a new code **`kindoo-no-description`**, `severity: 'review'`, **no action**.
- A defensive **"resolved segments but no primary"** fallback emits `kindoo-unparseable` as **`review`** (no action). The code review (`84faeb4`) reverted this from the earlier `drift` cut: there the Description *did* parse (it carries scope + parens, e.g. `Maple Ward (Bishop)`), so routing it to Update SBA would send that whole string as the calling and corrupt the seat.

**Invariant: a `review`-severity row never renders an action** — `fixActionsFor` returns no buttons for any review row, regardless of code (a top-of-function guard, `84faeb4`). The display-only Sync rows are therefore: `kindoo-no-description` (blank Description), a non-Guest present-but-unparseable `kindoo-unparseable` row, and the defensive parsed-but-no-primary fallback.

Shared types gained `KindooUnparseablePayload` (`{ memberEmail, calling }`) and a `{ code: 'kindoo-unparseable' }` member on the `SyncApplyFixInput` union, re-exported from `packages/shared/src/types/index.ts` + `packages/shared/src/index.ts`. `SYNC_DISCREPANCY_CODES` (`packages/shared/src/systemActors.ts`) gained `'kindoo-unparseable'` so the seat write stamps `SyncActor:kindoo-unparseable` and renders with the automated-actor chip. `kindoo-no-description` deliberately has **no** payload and is **not** in `SYNC_DISCREPANCY_CODES` — it never reaches the callable.

## Why

Sync converges drift between SBA and Kindoo in one direction (Kindoo authoritative, locked PR #183). A `review`-severity row that offered no button was a genuine dead end after the Kindoo-write path was removed: the operator saw drift but had no way to act on it in the panel. The fix derives an SBA-side action wherever one is safe to derive, and keeps `review` only where it isn't.

A **present** but unparseable Description carries intent — calling text that doesn't fit `Scope (Calling)` — read as "a church-wide (stake-scope) calling," so the action moves the seat to stake scope and stores the raw text as its calling. But "actionable" is the wrong default in three situations the review surfaced: on a **foreign site** the stake-scope move is meaningless (stake-scope ⇒ home); on a **non-Guest** (a Kindoo Manager whose own Description happens to be unparseable) an Update SBA would overwrite a manager's real seat; and for an **already-aligned** seat there's no drift to act on, so the row must self-resolve like every other drift code rather than persist. Hence the home-site / Guest / unaligned gating, with non-Guest and the can't-act fallback demoted to display-only `review` and the review-guard invariant enforcing "review ⇒ no action."

The earlier doc cut (commit `2e1b3f5`) claimed an unparseable calling *never* grants app access. The code review corrected this: "unparseable" only means the Description didn't match `Scope (Calling)`, but a bare template name (no parens, e.g. `Stake Clerk`) still reaches this path and *is* a real `give_app_access` stake template. Dropping its access on the stake move would be a silent regression, so the handler now reshapes the access doc in one write — old scope reaped, `importer_callings['stake']` written iff the calling matches a stake template, doc deleted when the final importer map and `manual_grants` are both empty. `kindoo_site_id` is cleared on the same move because stake-scope primaries resolve to the home site (spec §15 Phase 1); a Sync run against a foreign site could otherwise leave a foreign id stranded on a now-stake-scope seat.

## What didn't change that you'd expect to

- **The `extra-kindoo-calling` backend path is byte-for-byte unchanged.** Only the button label and the discrepancy's severity moved; the append-to-`callings[]` callable and the dispatch `testId` (`add-callings-sba`) are the same.
- **A non-template `kindoo-unparseable` calling still grants no app access.** Only a `give_app_access` **stake**-template match earns `importer_callings['stake']`; a genuinely free-text church-wide calling reaps the old scope and writes no new grant. App access for such a calling is a deliberate manual grant, not a Sync side-effect.
- **The Sync read/detection path is otherwise intact.** Parser, derived-buildings enrichment, active-site scoping, and every other discrepancy code are unchanged.
